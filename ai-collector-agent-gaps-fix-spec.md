# مواصفات تقنية: إغلاق ثغرات وكيل التحصيل (ai-collector-agent.ts)

**الهدف من هذا الملف:** تسليمه مباشرة لـ Claude Code لتنفيذ التعديلات على الوكيل. كل بند فيه يحتوي: المشكلة، التصميم المطلوب، ومخرجات/بيانات محددة (Schemas) عشان التنفيذ يكون حرفياً وغير قابل للتفسير المفتوح.

**ملاحظة مهمة:** بند إثبات الهوية (\#1) يلمس جمع بيانات شخصية حساسة (آخر 4 أرقام هوية). يُفضّل مراجعة سريعة مع أي طرف مسؤول عن الامتثال/حماية البيانات لديك قبل الإطلاق، حتى لو كان التصميم التقني هنا سليماً.

---

## 0\. البنية العامة: Pipeline حتمي بترتيب أولوية ثابت

المشكلة الحالية: لا يوجد ترتيب موثّق بين الـ Guards، فلو تحقق شرطان معاً يصير السلوك غير متوقع.

**الحل:** فصل صريح بين مرحلتين، بترتيب تنفيذ ثابت داخل كل مرحلة (أول guard يتحقق شرطه يوقف السلسلة \- short-circuit):

PRE-MODEL PIPELINE (قد توقف استدعاء النموذج كلياً):

  1\. crisis\_distress\_gate          \[جديد\]

  2\. stop\_contact\_gate              \[جديد\]

  3\. identity\_verification\_gate     \[جديد\]

  4\. multi\_portfolio\_clarification\_gate  \[معدّل\]

  5\. repeated\_question\_guard         \[قائم\]

  → استدعاء النموذج (إن لم يتوقف أي guard أعلاه)

POST-MODEL PIPELINE (تنقّي رد النموذج قبل الإرسال):

  1\. grace\_period\_guard              \[قائم\]

  2\. premature\_dispute\_guard          \[قائم\]

  3\. promise\_gate (hasTemporalRef)    \[معدّل\]

  4\. denial\_guards\_f\_g                \[قائم\]

  5\. self\_introduction\_consistency\_check \[معدّل\]

  6\. deflection\_to\_management\_guard    \[قائم\]

  7\. multi\_debt\_listing\_guard          \[قائم\]

  8\. info\_request\_pressure\_strip       \[قائم\]

  9\. anti\_repetition\_guard             \[معدّل\]

كل guard يرجّع نتيجة من نوع:

type GuardResult \=

  | { action: "pass" }

  | { action: "short\_circuit"; replyText: string; logReason: string }

  | { action: "mutate"; mutatedReply: string; logReason: string };

بهذا الشكل، أي إضافة guard جديدة بالمستقبل تُحدّد مكانها بدقة بدل التخمين.

## 1\. حارس إثبات الهوية (Identity Verification Gate) — جديد، الأهم

**المشكلة:** الرقم يُطابق ملف العميل تلقائياً، لكن لا يوجد تحقق فعلي أن الشخص الذي يرد على واتساب هو صاحب الدين. أي إفشاء لمبلغ/اسم جهة دائنة لرقم خاطئ \= خرق خصوصية محتمل.

**التصميم:**

- حقل جديد على مستوى المحادثة: `verification_status: "unverified" | "verified" | "locked"`  
- عند أول رسالة في محادثة جديدة (أو بعد X يوم من الخمول — قابل للتعديل، افتراضي 30 يوم):  
  - **يُسمح** بالرد على: تحية عامة (GREETING)، والتعريف الذاتي الثابت ("من أنت") — لأنها لا تكشف بيانات العميل.  
  - **يُمنع** كلياً: ذكر `creditor_name`، `current_balance`، أي رقم مرجعي/حساب — حتى لو طلبها العميل مباشرة.  
  - الوكيل يطلب عنصر تحقق واحد فقط من حقل موجود مسبقاً بملف القضية، مثل: آخر 4 أرقام من رقم الهوية/الإقامة المسجّل، أو تاريخ الميلاد. **لا تُستخدم بيانات يسهل تخمينها (مثل الاسم الكامل).**  
- المقارنة تتم **كودياً** (exact match على الحقل المخزّن)، أبداً لا تمر بحكم النموذج.  
- حد أقصى للمحاولات: **2**. بعد الفشل الثاني → `verification_status = "locked"` \+ تحويل فوري لطابور بشري \+ تجميد أي رسائل تحصيل تلقائية لهذي القضية حتى تدخّل بشري.  
- بعد النجاح: `verification_status = "verified"`، ويُحفظ لمدة الجلسة (لا يُعاد السؤال كل رسالة).  
- **سجل تدقيق إلزامي:** كل محاولة تحقق (وقتها، نتيجتها) تُخزّن لأغراض الامتثال.

interface VerificationAttempt {

  caseId: string;

  timestamp: string;

  fieldChallenged: "national\_id\_last4" | "date\_of\_birth";

  success: boolean;

}

## 2\. حارس "توقف عن التواصل" (Stop-Contact / Opt-Out Gate) — جديد

**المشكلة:** لا توجد آلية صريحة لطلب إيقاف التواصل.

**التصميم:**

- كشف نية الإيقاف عبر قائمة عبارات \+ تصنيف نية مبسّط (لا حاجة لنموذج كامل): "وقف الرسائل"، "لا تتواصلوا معي"، "أبلغ الجهة المختصة"، "هذا تحرش"، إلخ.  
- عند التحقق:  
  1. `contact_opt_out = true` \+ `timestamp` \+ `triggering_message` تُخزَّن على القضية.  
  2. إيقاف **فوري وكودي** (لا اعتماد على النموذج) لكل الرسائل الصادرة التلقائية المستقبلية لهذي القضية.  
  3. رد واحد ثابت للتأكيد فقط، بدون أي محاولة إقناع إضافية.  
  4. تحويل لطابور بشري لمتابعة القضية حسب سياسة الشركة.  
- **Guard إجباري بعد النموذج:** إن كان `contact_opt_out = true`، أي رد توليدي من النموذج يُستبدل كلياً — هذا الحارس له أولوية فوق كل الحراس الأخرى ولا يُسمح بأي استثناء.

## 3\. حارس الضغط النفسي الحاد (Severe Distress Gate) — جديد

**القرار المعتمد:** تحويل فوري للإدارة \+ رسالة تهدئة قصيرة (بدون استمرار الحديث عن الدين).

**التصميم:**

- كشف مبكر (Pre-model) لعلامات ضيق نفسي حاد مرتبط بالدين — قائمة عبارات \+ (اختياري) استدعاء تصنيف خفيف. هذا الحارس **يسبق كل شيء** في الـ pipeline، بما فيه التحقق من الهوية.  
- عند التحقق:  
  1. **لا يُستدعى النموذج التحصيلي إطلاقاً** لهذي الرسالة.  
  2. رد واحد قصير، هادئ، بدون أي طابع تحصيلي، يُعلن تحويلاً فورياً لشخص من الفريق — بدون استمرار بالموضوع.  
  3. `case_status = "urgent_human_review"`, `reason = "distress_signal"` — وتجميد كل رسائل المتابعة التلقائية حتى يُلغي إنسان هذا التجميد يدوياً.  
  4. تسجيل كامل نص الرسالة \+ الوقت لمراجعة الفريق البشري.  
- **مهم:** هذا الحارس "قاطع دائرة" فقط (circuit breaker) — وليس نظام تدخّل أزمات. لازم يكون عندكم إجراء بشري واضح خلفه: مين يستقبل التنبيه، وفي خلال كم دقيقة، وهل فيه تصعيد لجهة مختصة عند الحاجة. هذا الجزء التشغيلي مسؤولية فريقكم، لا الوكيل.

## 4\. تعديل: بوابة تعدد المحافظ (Multi-Portfolio Gate)

**المشكلة الحالية:** تتجاوز النموذج كلياً، فلو الرسالة فيها نية ثانية مدمجة (مثال: "متى موعد سداد الراجحي، وهل ممكن تقسيط؟") تُفقد النية الثانية.

**التصميم المعدّل:**

- عند الغموض (أكثر من محفظة ولم يُحدَّد أيها)، يُخزَّن:

pendingClarification \= { originalMessage: string, missingField: "company" }

- يُرسَل سؤال التوضيح كالمعتاد (بدون استدعاء النموذج — يبقى توفير التكلفة).  
- في الرد التالي من العميل: تُدمَج رسالته الجديدة مع `originalMessage` المخزّنة قبل تمريرها للنموذج، عشان أي نية ثانية بالرسالة الأصلية لا تُفقَد.

## 5\. تعديل: حارس التعريف الذاتي (Self-Introduction Guard)

**المشكلة الحالية:** استبدال الرد كاملاً بنص ثابت يكسر الطابع الطبيعي للحوار، خصوصاً وسط نزاع متقدم.

**التصميم المعدّل (حقن لا استبدال):**

- يُحقَن في تعليمات الجولة الحالية فقط: `must_state_fact_verbatim: "<النص الثابت لمن أنت>"` ويُطلَب من النموذج تضمين هذي الحقيقة بصياغة طبيعية متماشية مع السياق.  
- **Post-model check:** يتحقق أن المحتوى الوقائعي (الاسم، الصفة، الجهة) موجود فعلياً في رد النموذج (مطابقة نصية مرنة، لا حرفية تماماً).  
- فقط إذا فشل التحقق (الحقيقة غائبة أو محرَّفة) → يُستبدل الرد كاملاً بالنص الثابت كخطة بديلة (fallback)، لا كسلوك افتراضي.

## 6\. تعديل: حارس الوعد الزمني (Promise Gate / hasTemporalRef)

**المشكلة الحالية:** الاعتماد على كشف بسيط (regex/كلمات) قد لا يغطي تنوع اللهجة السعودية في التعبير عن الوقت.

**التصميم المعدّل (طبقتان):**

1. **قاموس صريح** (lexicon) مصنّف بفئات، يُبنى ويُحدَّث من محادثات حقيقية فعلية:  
   - نسبية: "بكرة"، "بعد بكرة"، "الأسبوع الجاي"  
   - مرتبطة بدخل: "بعد الراتب"، "أول الشهر"  
   - تواريخ صريحة: أي تاريخ/يوم مذكور حرفياً  
   - مناسبات: "بعد العيد"، "بعد رمضان"  
2. **استخراج بنيوي احتياطي** (فقط إذا لم يطابق القاموس ولكن السياق يحمل لغة التزام، مثل وجود فعل "أسدد/أحوّل" مع تعبير وقت غامض): استدعاء مخرجات منظَّمة صارمة:

{ "has\_temporal\_ref": true, "resolved\_date": "2026-07-15", "confidence": 0.86 }

- يُسجَّل الوعد فقط إذا: `has_temporal_ref = true` **و** `confidence` فوق حد معيّن **و** `resolved_date` ضمن سقف 30 يوم (الحارس القائم).  
- **حلقة تحسين مستمرة:** كل حالة يفوتها القاموس وتُكتشف بالطبقة الاحتياطية تُسجَّل لإضافتها للقاموس لاحقاً.

---

## 7\. تعديل: تعارض حقول المبلغ (current\_balance مقابل original\_amount)

**المشكلة:** لو ملف القضية فيه أكثر من حقل رقمي (مبلغ أصلي \+ رصيد حالي بعد سداد جزئي)، خطر أن يقتبس النموذج الحقل الخطأ دون أن "يخالف" القاعدة المكتوبة ظاهرياً.

**الحل: تطهير السياق قبل الإرسال للنموذج (Canonicalization) — لا حل نصّي، حل بنية بيانات:**

interface CaseSummaryForPrompt {

  creditorName: string;     // من creditor\_name أو portfolio\_name فقط

  amountDue: number;        // \= current\_balance دائماً، بدون استثناء

  hasPartialPayment: boolean; // علم فقط، بدون رقم

}

- **لا يُحقَن `original_amount` أو أي حقل رقمي آخر في الـ prompt إطلاقاً**، إلا في مسار فرعي مخصص لو العميل سأل تحديداً "ليش المبلغ تغيّر؟" — وفي هذي الحالة فقط، تُحقَن تفاصيل واضحة المسمّيات في رسالة نظام منفصلة لهذا السؤال تحديداً.  
- بهذا الشكل المشكلة تُحل من مصدرها: النموذج لا يرى أصلاً حقلاً لا يُفترض أن يقتبسه.

## 8\. تحسين: قائمة الردود البشرية (Anti-Robotic Guard)

**المشكلة:** قائمة ثابتة واحدة قد تتكرر هي نفسها مع مرور الوقت/العملاء.

**التحسين:** تجميع ≥15 صيغة لكل (نية × سياق)، مع تتبّع آخر الصيغ المستخدمة لكل جلسة (`usedVariantIds`) وعدم تكرار نفس الصيغة داخل الجلسة نفسها.

## 9\. القرار النهائي: ترشيح النموذج

**النموذجان المعتمدان (مزوّد واحد — Anthropic):**

| المستوى | النموذج | الاستخدام |
| :---- | :---- | :---- |
| المستوى 1 — الحجم العام | `claude-haiku-4-5-20251001` | GREETING, INTRODUCTION, INFO\_REQUEST, GENERAL |
| المستوى 2 — التصعيد | `claude-sonnet-4-6` | DISPUTE, NEGOTIATION، وأي حالة يرفعها deflection-to-management guard دون أن تستوجب تدخلاً بشرياً فعلياً |

**سبب الاختيار:**

- مزوّد واحد \= SDK واحد، تسعير واحد، دلالات tool-calling متطابقة عبر كل الكود — يبسّط تسليم التنفيذ الكامل لكلود كود دفعة واحدة بدل التعامل مع 3 تنسيقات API مختلفة.  
- بنية الـ Guards الحتمية (§0–§8) تستوعب أخطر القرارات كودياً (المهلة، الوعد، اختراع البيانات)، فهامش الفرق بين مزوّدين مختلفين على الانضباط بالقواعد أقل أهمية هنا من تكلفة الصيانة الهندسية لخلط أكثر من مزوّد.  
- التدرّج Haiku → Sonnet يربط مباشرة بجدول النيات (Intent) الموجود أصلاً بالوكيل، بدون أي منطق توجيه إضافي يُبتكَر من الصفر.

**منطق التوجيه (Routing Logic) — جاهز للتنفيذ:**

function selectModel(intent: Intent): "claude-haiku-4-5-20251001" | "claude-sonnet-4-6" {

  const escalationIntents \= \["DISPUTE", "NEGOTIATION"\];

  return escalationIntents.includes(intent)

    ? "claude-sonnet-4-6"

    : "claude-haiku-4-5-20251001";

}

**ملاحظة للمستقبل:** لو لاحظتم بعد الإطلاق وبحجم حقيقي من المحادثات أن Haiku ضعيف على اللهجة لحالة معيّنة، الحل تعديل سطر واحد في `selectModel` لرفع ذاك الـ Intent تحديداً لـ Sonnet — بدون أي تغيير بنيوي في الوكيل أو الـ Guards.

## ترتيب تنفيذ مقترح لكلود كود (لو حبيت تسليم على مراحل)

1. حارس إثبات الهوية (§1) — أمني/قانوني، الأولوية القصوى  
2. حارس الضغط النفسي الحاد (§3)  
3. حارس توقف التواصل (§2)  
4. إعادة هيكلة ترتيب الـ Pipeline (§0) — تنظيم بدون منطق جديد  
5. تطهير حقول المبلغ (§7)  
6. تحسين قاموس الوعد الزمني (§6)  
7. تعديل بوابة تعدد المحافظ (§4)  
8. تعديل حارس التعريف الذاتي (§5)  
9. تحسين قائمة الردود البشرية (§8)
